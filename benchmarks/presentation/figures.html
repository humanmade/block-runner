The path from any HTML to clean native blocks

What runs when an end user pipes HTML through the CLI. One pass: translate the structure, prove it against real Gutenberg, fill in real media — and surface anything it couldn’t express natively.

Read it left to right. The gate runs on every conversion, so “valid” means what the editor means. Nothing degrades quietly: each fallback or deviation is reported with the exact input line that caused it.

The benchmark loop

One ideal answer per layout; many producers supply HTML; the engine converts; a deterministic scorer grades the result against the ideal tree; every run is logged with provenance so progress is auditable.

Why it’s trustworthy. The scorer is deterministic (no LLM judge), scores are graduated (partial credit per axis), and a suiteHash ties each result to the exact suite — so a score change is attributable to the engine only when the suite is unchanged. The dashed backtest loop re-runs older engine versions against today’s fixed suite, producing a true apples-to-apples progress curve.

How a score is built. Per fixture, five axes: STRUCT (right blocks, right nesting, right order) · CONTENT (right text/media in the right block) · VALID (passes the headless-Gutenberg gate; invalid halves the score) · FALLBACKS (every core/html blob is a miss) · COVERAGE (input text that survived). Composite = 0.75·STRUCT + 0.25·CONTENT, halved if invalid.

Honest caveats. The raw-LLM paths (“Claude alone”, “Codex alone”) write the markup themselves, no Block Runner assembly — strong structurally but capped by validity (right tree, invalid save-markup → the gate halves it). They’re single non-deterministic runs; Block Runner’s LLM numbers are best-ever baselines. Engine A (rules) is free, offline, deterministic but brittle.
